# RunPod — 7 возможностей для повышения эффективности

Фундаментальные возможности, которые открывает доступ к мощному облачному железу (RunPod CPU pod 32 vCPU / 124 GB RAM) по сравнению с текущим prod VPS (3 vCPU / 8 GB). Переход от точечных прогонов к систематическому research-циклу.

## 1. Систематический grid search по параметрам стратегии

Сейчас workflow: ретрейн каждые 4h на rolling 72h, одна модель с зафиксированными параметрами. С подом можно **один раз** найти глобальный оптимум и закрепить:

- TP/SL пары (~50 комбинаций)
- Confidence threshold (~10 шагов)
- Partial TP progress
- Trailing SL шаги
- Длина окна CNN, horizon
- Hyperparameters моделей: глубина XGBoost, learning_rate, embedding size CNN

**Оценка**: ~500 бэктестов. На prod VPS = 2 недели. На поде = 1 час.

## 2. Монте-Карло вместо единичных точек

Текущий бэктест возвращает **одно число** (Net-WR, PF, max DD). Это одна реализация из распределения — зависит от seed, post-only-reject, fill timing. С подом можно прогнать 500 бэктестов с разными seed'ами и получить **доверительные интервалы**:

- Не "WR=41.7%", а "WR=41.7% ± 3.2%, 95% CI [35.3, 48.1]"
- Позволяет отличить робастную стратегию от случайно везучего прогона

**Что меняет**: переход от "кажется работает" к "доказано что работает в 95% реализаций".

## 3. Walk-forward с rolling окном

Сегодняшний 80/20 split — это **один** тест. Rolling walk-forward даёт **N независимых out-of-sample оценок**:

- Train на 3 дня → test на 1 день → сдвиг на 1 день → repeat
- Единственный честный способ проверить, что модель не переобучилась
- На неделе данных получаем 7 оценок вместо 1

**Сейчас нереально**: неделя данных требует 1+ час на prod VPS. На поде — минуты.

## 4. Feature ablation / importance analysis

Сейчас 34 ручных фичи + 64 из CNN = 98-dim вход. **Какие реально работают?**

- Систематически выключать каждую фичу, мерить деградацию метрик
- Kick features which contribute <0.5%% к PF
- Результат: либо подтверждение всех 34, либо сокращение до 10-15

**Эффект**: быстрее inference на prod, меньше overfitting, проще debugging.

## 5. Портфель инструментов

Сейчас торгуем только BTCUSDT. Пайплайн одинаковый для любого perpetual'а:

- ETHUSDT, SOLUSDT, BNBUSDT — добавить по одной паре на попытку
- Тренируется параллельно — 4 инструмента за те же 2.5 минуты
- Диверсификация PnL, независимые источники сигналов
- Возможна кросс-инструментная корреляция как новая фича

**Путь**: сначала доказать alpha на BTC, потом расширять.

## 6. Большие модели

CNN сейчас крошечный — 2 conv-слоя, 3×20×50 вход, 64-dim embedding. С 124 GB RAM и 32 vCPU возможно:

- **Глубже CNN** — больше слоёв, dilated convolutions для больших контекстов
- **Transformer-голова** поверх multi-stream данных (BTC + ETH + 4 биржи как токены)
- **Ансамбль из 20 моделей** вместо 5 (больше bagging-сидов, XGBoost с 10k деревьев)
- **Knowledge distillation** — обучить большую модель на поде, выдистиллировать в лёгкую для prod inference

## 7. Reinforcement learning для sizing

Сейчас размер позиции — статическая формула от voting majority:

- 5/5 votes → ×1.0, 4/5 → ×0.75, 3/5 → ×0.50
- Hurst uncertain → ×0.5 дополнительно

RL-агент, обученный на forward-sim'е (live_sim.simulate_trade), мог бы динамически адаптировать размер к режиму рынка и уверенности ансамбля. RL требует **тысячи эпизодов** — только на поде реально.

**Потенциал**: улучшение avg_win/avg_loss ratio без изменения базового сигнала.

## Что НЕ открывается (честные ограничения)

- **Скорость live-execution** — упирается в 3ms до Binance, не в compute
- **Качество данных** — дыры в depth из-за сетевых проблем compute не лечит
- **Силу сигнала** — если в микроструктуре BTC нет alpha на нашем горизонте, железо её не создаст
- **Риск переобучения растёт** — больше экспериментов = больше шансов найти "победителя", который артефакт. Нужна железная out-of-sample дисциплина (walk-forward + paper trading)

## Приоритет внедрения

1. **Rolling walk-forward на чистых 3-5 днях** (после 2026-04-15) — проверить стабильность PF между прогонами
2. **Grid search по TP/SL и confidence threshold** — текущий прогон показал одну точку, возможно не оптимальную
3. **Feature ablation** — возможно 3-5 фич из 34 делают 80% работы
4. **Затем** — портфель инструментов и RL-sizing. Сначала выжать максимум из BTC.

Без compute-ограничений переходим от "угадывать на глаз" к "систематически доказывать".
